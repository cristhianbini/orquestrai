# R6-16.1 — Contagem bruta de string não mede debug real

## Contexto

O R6-16 removeu 12 pontos de debug mapeados manualmente (leitura visual do
`dashboard.html`). Uma auditoria pós-commit encontrou que **3 dos 12** ainda
existiam em produção: `B101` (bloco inteiro nunca entrou na lista), e uma
segunda ocorrência cada de `B163` e `B288` (a lista tinha só 1 ocorrência de
cada, mas existiam 2).

## Causa raiz

Duas falhas distintas, mesma origem: **leitura visual rápida de um arquivo
de ~4700 linhas erra por omissão**, não por má vontade.

1. **Nomes parecidos colidem na memória de curto prazo.** A lista original
   continha `B100` (2 ocorrências, hook xterm + scroll) e `B101` (atalhos de
   teclado) como itens vizinhos. `B101` nunca foi incluído na lista final de
   âncoras — provavelmente fundido mentalmente com `B100` na hora de
   consolidar a lista.
2. **Contagem por string não distingue ocorrência única de múltipla.**
   `B288` tinha 2 `console.log` diferentes no arquivo; a lista mapeou só 1.

## O padrão generalizável (o que vale levar para o futuro)

> **Contagem de string bruta (`grep -c "TAG"`) mede menções, não debug.**
> Uma tag pode aparecer em: `console.log` (debug real), nome de variável,
> classe CSS, comentário, id de elemento, ou até em código funcional
> (ex.: `B94` é usado tanto em debug quanto no sistema de confirmação real
> `window.__b94Confirm` — remover por contagem bruta teria apagado uma
> feature de segurança).

Isso já apareceu antes nesta base de código sob outro nome: a lição
`R6-13.5.3` ("nome enganoso não é código morto") tratava do mesmo princípio
para funções; a auditoria de cobertura de auth ("contar rotas totais vs.
rotas com header Authorization") é a mesma ideia aplicada a segurança.
**É um padrão recorrente, não um incidente isolado — vale desconfiar sempre
que uma decisão de remoção/segurança depender só de contagem de string.**

## Regra prática adotada a partir daqui

Antes de fechar qualquer lista de remoção em massa de debug:

```bash
# 1. contagem bruta (informativa, NAO decisiva)
grep -c "TAG" arquivo

# 2. contagem filtrada por comportamento real (decisiva)
grep -n "console\.\(log\|warn\|error\)" arquivo | grep -- "TAG"

# 3. se a contagem bruta > contagem filtrada, ler o contexto de CADA
#    ocorrencia bruta restante antes de declarar "seguro, nao e debug"
```

Se a contagem filtrada bater com o número de itens já removidos, está
completo. Se sobrar, há gap — investigar antes de fechar a rodada.

## Resultado da correção (CTXR616GAP01)

- `B101`: bloco continha um hint de onboarding real (`if(!seen){...}`)
  misturado com debug puro (`window.__B101` + `console.log`). **Preservado
  o hint, removido só o debug** — outro caso de "nome enganoso ≠ tudo é
  descartável".
- `B163`: preservado `es.close()`, removido só o `console.log` de fluxo.
- `B288`: segunda ocorrência removida.
- Validado: `node --check` em todos os blocos + console limpo confirmado
  visualmente (Ctrl+Shift+R). Commit `1b02515`.
