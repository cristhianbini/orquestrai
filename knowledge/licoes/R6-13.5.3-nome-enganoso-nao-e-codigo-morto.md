# Licao R6-13.5.3 — Nome enganoso nao prova codigo morto

## Contexto
Na limpeza de legado do dashboard.html (strangler fig), as funcoes
`masxSet` / `masxCard` / `masxResetAll` pareciam codigo morto: o prefixo
`masx` remetia ao container `.masx-card` / `#masx-cards`, ja removido em
R6-13.5.x. A descricao herdada do chat anterior dizia literalmente
"codigo morto que roda".

## A armadilha
`masxSet` NAO era morto. A primeira linha dele chamava:

    function masxSet(ag, state, msg){
      b124g5Apply(ag,state);        // <- VIVO: anima os cards reais .oq46y-card
      var c=masxCard(ag); if(!c) return;   // <- daqui pra baixo, morto (.masx-card)
      ...
    }

`b124g5Apply` e o caminho VIVO da animacao: borda pulsante `b124g5-working`
-> anel `b124g5-ready`, dirigido pelo `PIPE` no handler SSE. So o "rabo"
`.masx-card` (apos `if(!c)return`) era morto.

Remover a funcao pelo nome teria matado a animacao viva do pipeline MAS.
Pior: seria uma **regressao silenciosa** — o bloco LAVE ainda seria gerado,
entao passaria em qualquer teste superficial. So apareceria como "as bordas
nao acendem mais", dificil de rastrear depois.

## Regra
Nunca decidir remocao pelo NOME do simbolo. Ler o CORPO inteiro e rastrear
o que ele chama. Um simbolo pode ter sido reaproveitado e carregar logica
viva sob um nome legado/enganoso.

## O que salvou (metodo aplicado)
1. `grep -rln` no repo: confirmou `masx*`/`b124g5Apply` confinados a
   `src/dashboard.html` (nenhum island React dependia).
2. Leitura do corpo, nao do nome: revelou a chamada a `b124g5Apply`.
3. Script de remocao ancorado por CONTEUDO (abort se ancora != 1x),
   testado contra o arquivo real antes de rodar em producao.
4. `node --check` nos 74 blocos <script>.
5. Prova viva pos-corte: MutationObserver em producao mostrou o pipeline
   `working -> ready` andando pelos 8 cards. So entao commit.
